# 安全与权限边界

## 安全目标

该产品面向明确授权的 Windows 远控场景。当前已实现的是临时协助 / TOTP 模式，后续会增加可显式启用的常驻远控模式；两种模式都必须满足以下安全目标：

- 用户明确知道 host 当前是否正在运行，以及是否处于可被连接状态。
- 用户可以通过关闭窗口、结束会话或禁用常驻模式随时终止控制。
- 用户可以在被控端的可见界面中实时看到远控操作摘要和当前模式状态。
- 客户可以直接转发被控端自动复制的连接信息，但其中不得包含控制端凭据。
- 未持有控制端 token 的人不能发起控制。
- 临时模式下，未获得客户当前 TOTP 的人不能建立会话。
- 长期模式下，未获得有效长期凭据的人不能建立会话。
- 服务器重启或被控端退出后，会话自然失效。

## 鉴权模型

被控端上线不需要 token。客户只需要能连接到服务器，就可以登记在线、显示机器 ID/TOTP，并把连接信息发给研发。

当前已实现的临时协助模式使用两层鉴权：

1. 控制端 token：证明调用者是授权研发或授权自动化系统。
2. 机器 ID + TOTP：证明用户当前正在配合本次临时支持。

控制端 token 由服务端配置，只用于控制端，不嵌入控制端二进制，也不配置到被控端。TOTP seed 由被控端每次运行随机生成，关闭窗口后失效。

后续长期配对模式会在此之外增加长期凭据或配对关系，但不会把长期凭据写入剪贴板、普通日志、审计摘要，也不会把 session token 变成长期静态凭据。

机器 ID 必须稳定但不能泄露原始机器标识；当前实现约束见 [windows-apis.md](windows-apis.md)。

## TOTP

- TOTP seed 每次被控端启动时随机生成。
- TOTP 周期默认 120 秒，允许通过配置调整。
- 显示 6 位数字。
- 被控端本地验证 TOTP。
- 可接受当前窗口和相邻一个时间窗口，以容忍轻微时钟偏差。
- 控制端和被控端必须使用同一 TOTP 周期；不一致时不得建立会话。

## 长期凭据约束

- 固定密码、私钥、恢复码或类似长期凭据不能进入剪贴板、普通日志、审计摘要或 crash 输出。
- Windows 侧长期凭据应优先使用 DPAPI、Credential Manager 或等价本机安全存储。
- 固定密码方案需要速率限制、锁定策略和明确错误返回，避免在线爆破。
- 密钥对方案需要定义设备指纹、信任首次连接、撤销和轮换流程。
- 必须区分“host 长期可发现或可连接”和“控制端已经取得当前 session”这两个状态。

## 会话安全

- 会话 token 由服务端生成，必须使用安全随机数。
- 会话 token 只保存在控制端本地会话文件中。
- 被控端同一时间只允许一个会话。
- 被控端断开、退出或显式关闭会话后，服务端必须吊销 session。
- 服务端重启后所有 session 失效。

## 操作可见性

当前控制台，以及后续若增加 GUI、托盘或其他可见界面时的对应状态面板，必须显示：

- 是否有 active session。
- 当前使用的是临时协助模式还是长期配对模式。
- 当前是否处于长期可连接状态。
- 剪贴板连接信息复制成功或失败状态。
- 是否已阻止系统休眠和显示器熄屏。
- 最近一次命令类型。
- 最近一次命令时间。
- 实时追加的操作审计摘要。
- 当前权限状态：普通用户或管理员权限。
- 关闭窗口可终止控制，或禁用常驻模式可撤销长期连接能力的提示。

如果被控端检测到当前进程是管理员权限，必须使用醒目颜色高亮显示。该提示不是为了鼓励提权，而是为了让客户和研发明确当前远控动作会拥有更高系统权限。

当前交互不要求客户逐项确认每条命令，因为控制端面向高效的 agent 辅助诊断。但它仍然不得增加隐藏窗口、不可见后台控制或其他绕过客户感知的行为。

## 权限边界

被控端只拥有当前进程用户权限。

- 若客户以普通用户运行，则命令也是普通用户权限。
- 若需要管理员权限，客户必须手动以管理员身份运行被控端。
- 产品不实现自动 UAC 提权或 UAC 绕过。
- 产品不安装驱动。
- 产品不注入其他进程。
- 产品不修改系统电源计划；仅在被控端进程运行期间发起临时电源请求。

## 网络安全

- 生产环境必须使用 WSS。
- 推荐部署在 443 端口后。
- 控制端 token 应至少 128 bit 随机强度。
- 被控端连接 `/ws/host` 不要求 token，但服务端应对主机登记做基础速率限制，避免恶意刷在线主机。
- 服务端日志不得记录完整 token、session token 或 TOTP seed。
- 认证失败应做基础速率限制，避免暴力尝试。

## 文件安全

- 文件上传默认不覆盖已有文件，除非传入 `--overwrite`。
- 文件传输后必须校验 SHA-256。
- 路径必须按 Windows 规则解析，拒绝空路径和非法路径。
- 下载敏感文件由研发自己判断授权范围，工具层不做内容审查。

## 审计

当前审计模型使用三端结构化日志，不引入中央审计数据库。每个操作都必须能通过 request ID 在 host、controller 和 server 日志中对齐。

三端职责：

- 被控端：写入实际执行日志，并在控制台实时显示操作摘要。
- 控制端：写入操作发起日志，记录 CLI 子命令、参数摘要、输出文件路径和结果。
- 服务端：写入中继日志，记录认证、会话、命令类型、消息转发和断开原因。

应记录：

- 主机上线和下线。
- 会话创建和关闭。
- 命令类型。
- 命令开始和结束时间、request ID、task ID 和 duration。
- 命令是否成功。
- 传输文件大小、已传输字节数和哈希。
- 文件路径 basename 或归一化路径摘要。
- TCP tunnel 的方向、listen/target 端点摘要、tunnel ID 和 stream ID。
- `exec` 的 program basename、argv 数量、timeout 和工作目录 basename。
- `keyboard.type` 的字符数和字节数。
- 当前被控端权限状态。
- request ID 和 session ID。

不应记录：

- 完整控制端 token。
- session token。
- TOTP seed。
- 固定密码、私钥、配对 secret 或其他长期凭据原文。
- 剪贴板完整内容。
- 默认完整命令输出。
- `keyboard.type` 输入全文。
- 默认完整 `exec` argv 明文。
- 默认完整用户文件路径。
- 文件内容。

任何 `token`、`password`、`passwd`、`secret`、`key`、`*_key` 等疑似敏感赋值字段都应写成 `[redacted]`。路径型字段默认只保留 basename；需要完整路径时应通过明确的后续授权流程临时采集，不能作为 host audit 的默认行为。

如果任一端本地日志写入失败，该端必须输出 warning。被控端的日志写入失败 warning 必须实时显示在控制台。

## 明确不做

- 不做静默控制。
- 不做隐藏进程。
- 不做隐藏窗口或不可见后台控制。
- 不做未经用户显式启用、不可撤销的常驻化或开机自启。
- 不做自动 UAC 提权或 UAC 绕过。
- 不做键盘记录。
- 不做后台屏幕录制。
